# Plan: Enhance Attendance Delete Endpoint with Optional Filters

## Current State
- `DELETE /attendance/{attendance_id}` deletes a single record by ID.
- Returns 404 if not found.
- No bulk deletion support.

## Requirements
1. Add optional query parameters for bulk deletion via `DELETE /attendance/`:
   - `date` (single date)
   - `date_from` + `date_to` (date range)
   - `service_type`
   - `contact_id`
2. If no filters provided on bulk route, delete **all** attendance records.
3. `DELETE /attendance/{attendance_id}` deletes single record by ID, returns 404 if not found.
4. Bulk delete (`DELETE /attendance?filters...`) always returns 200 with JSON body containing count.
5. Proper error handling and timezone (SAST) conversion where applicable.

## Proposed Design

### Proposed Design

#### Approach: Two Separate Route Handlers
Use two distinct delete endpoints to maintain REST semantics:
- `DELETE /attendance/{attendance_id}` → deletes single record, returns **404** if not found, **204 No Content** (or 200 with body) on success.
- `DELETE /attendance/` → bulk/delete-all, always returns **200 OK** with JSON body: `{"message": "...", "deleted_count": N}`.

**Rationale:** Single deletion by ID is a "find-and-remove" operation; absence of the resource is legitimately a 404. Bulk deletion is a "search-and-destroy" operation — zero matches is a successful result (nothing to delete), hence 200.

### Service Layer Changes (attendance_service.py)
- Keep `delete_attendance(self, attendance_id: int) -> bool` (returns True if deleted, False if not found).
- Add `delete_attendance_filtered(attendance_id=None, date_from=None, date_to=None, service_type=None, contact_id=None) -> int`:
  - Returns count of deleted rows.
  - If `attendance_id` is not None: delegates to `delete_attendance` and returns 0 or 1.
  - Else: builds filtered query, executes bulk `delete()`, commits, returns count.
  - If all filters are None: deletes all records.

### Router Changes (attendance.py)
1. Keep existing `@router.delete("/{attendance_id}")` handler (unchanged except response format).
2. Add new `@router.delete("/")` handler for bulk/delete-all.
   - Parameters: all optional query filters (`date`, `date_from`, `date_to`, `service_type`, `contact_id`).
   - Converts dates to SAST via `convert_to_sast`.
   - Calls `service.delete_attendance_filtered(...)` with all filters.
   - Returns `{"message": "...", "deleted_count": deleted_count}` with 200.
   - Message logic:
     - `deleted_count = 0` → `"No attendance records found matching the criteria."`
     - `deleted_count = 1` → `"Deleted 1 attendance record."`
     - `deleted_count > 1` → `"Deleted {deleted_count} attendance records."`
3. Consistent error handling: exceptions roll back and return 500.

### Single-ID Delete Response Options
- **Option A (Prefer):** `204 No Content` with empty body. Minimal; clients check status code (2xx = success, 404 = not found).
- **Option B:** `200 OK` with JSON `{"message": "...", "deleted_count": 1}` for consistency with bulk mode.
Both are valid; 204 is true RESTful for successful DELETE. If clients need feedback on count, use 200.

**Recommendation:** Use **204 No Content** for single-ID delete, **200 OK with JSON body** for bulk mode. Clear semantic distinction.

### Service Layer Changes (attendance_service.py)
- Keep `delete_attendance(self, attendance_id: int) -> bool` (returns True if deleted, False if not found).
- Add `delete_attendance_filtered(attendance_id=None, date_from=None, date_to=None, service_type=None, contact_id=None) -> int`:
  - Returns count of deleted rows.
  - If `attendance_id` is not None: delegates to `delete_attendance` and returns 0 or 1.
  - Else: builds filtered query (`Attendance.query.filter_by(...)`), executes bulk `query.delete(synchronize_session=False)`, commits, returns count.
  - If all filters are None: executes `db.query(Attendance).delete(synchronize_session=False)` to delete all.

### Router Changes (attendance.py)
1. Keep `convert_to_sast` helper.
2. Add stacked `@router.delete` decorators.
3. In function body:
   - If `attendance_id` is not None: call `service.delete_attendance(attendance_id)` (or new method with only ID) and get count (0/1).
   - Else:
     - Convert `date` → `date_from_dt` and `date_to_dt` covering entire day in SAST.
     - Convert `date_from`/`date_to` to SAST using `convert_to_sast`.
     - Pass filters to `service.delete_attendance_filtered`.
   - If no filters at all (attendance_id is None and all query filters are None): proceed to delete all (count = result of delete with no filter).
   - Commit and craft response message based on count.
4. Error handling: catch exceptions, rollback, return 500.

### Edge Cases
- `date` provided together with `date_from`/`date_to`: `date` takes precedence; range ignored.
- `date_from` without `date_to`: only lower bound applied.
- Invalid IDs → 0 count, return 200 with "No attendance records..." message.
- Concurrent modifications: handled by DB transaction.
- Timezone: `service_date` stored as timezone-aware; all comparisons use SAST.

### Safety Considerations
Deleting **all** records is a dangerous operation but is a stated requirement ("default deletes attendance for all the record"). It is protected by authentication (`get_current_active_user`). No additional confirmation required per user request.

### Files to Modify
1. `app/services/attendance_service.py` — add `delete_attendance_filtered`.
2. `app/routers/attendance.py` — enhance delete endpoint as described.

### Bulk Delete Handler – `DELETE /attendance/`
- Parameters (`Query`): `date`, `date_from`, `date_to`, `service_type`, `contact_id`.
- Date handling: If `date` provided, it becomes a full-day range (00:00:00 - 23:59:59) in SAST, overriding `date_from`/`date_to`. Otherwise, convert `date_from`/`date_to` to SAST.
- Calls `service.delete_attendance_filtered` passing converted values.
- Returns `{"message": "...", "deleted_count": N}` with **200 OK**.

### Single-Delete Handler – `DELETE /attendance/{attendance_id}`
- Existing handler: call `service.delete_attendance(attendance_id)`.
- If returns True → **204 No Content** with empty body.
- If returns False → **404 Not Found** with JSON error: `{"detail": "Attendance record not found"}`.

### Edge Cases
- `date` + `date_from`/`date_to`: `date` takes precedence; log warning about ignored range.
- `date_from` without `date_to`: only lower bound applied.
- Empty bulk result: return 200 with `"deleted_count": 0` and appropriate message.
- Timezone: `service_date` stored as timezone-aware; all comparisons use SAST.
- Concurrent modifications: handled by DB transaction; bulk delete is atomic.
- Delete-All protection: authenticated endpoint only; no separate confirmation per requirement.

2. `app/routers/attendance.py` — keep single-ID route, add new bulk route (`/`).
