# Plan: Prevent Attendance Duplicates

## Problem Analysis
- Attendance records are appearing with exact duplicates (same name, phone, location, member status)
- This causes issues in reporting and data integrity
- Current backend logic checks for existing attendance before insertion, but race conditions or multiple rapid requests could bypass this check
- No database-level constraint prevents duplicates
- Issue stems from Flutter app implementation sending duplicate requests

## Solution Overview
Implement database-level uniqueness constraint to prevent duplicate attendance records for the same contact, service type, and date. Update service logic to handle constraint violations gracefully.

## Implementation Steps

### 1. Update Attendance Model
- Add UniqueConstraint on (contact_id, service_type, service_date) to prevent duplicates
- This assumes multiple attendances per day for same service are not allowed (even with different times)

### 2. Create Database Migration
- Generate Alembic migration to add the unique constraint
- Handle existing duplicates if any (though unlikely based on current logic)

### 3. Update Attendance Service
- Modify record_attendance to catch IntegrityError from unique constraint violation
- Return appropriate error message instead of allowing duplicate insertion
- Ensure transaction rollback on failure

### 4. Testing
- Test duplicate prevention with API calls
- Verify existing functionality still works
- Check PDF generation handles unique records correctly

### 5. Client-Side Note
- Inform development team to fix Flutter app to prevent sending duplicate attendance requests
- Backend protection provides safety net

## Files to Modify
- app/models.py: Add UniqueConstraint to Attendance class
- app/services/attendance_service.py: Handle IntegrityError in record_attendance
- Alembic migration file: New migration for constraint

## Potential Risks
- Existing duplicate data in DB would cause migration failure (mitigate by cleaning data first if needed)
- Unique constraint might be too strict if same contact attends multiple sessions of same service on same day (review business logic)

## Validation
- After implementation, attempt to record duplicate attendance should fail with clear error message
- PDF reports should show unique records only
